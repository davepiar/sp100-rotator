## Architectural finding worth flagging

The PreToolUse hooks (`pre_order_check.py`, `market_hours_check.py`,
`drawdown_guard.py`) only fire on **Bash invocations whose command string
matches `curl -X POST .../v2/orders`**. They do NOT fire when
`scripts/execute_plan.py` posts via `requests.post(json=…)`, because there is
no Bash command in that flow.

Implication: every safety check in those hooks (universe membership, position
count, 5% notional cap, session turnover, market hours, drawdown guard) is
unenforced for `execute_plan.py`. Today the script trusts whatever is in the
signed-off plan.

Two ways to close the gap:
1. **Move checks into a shared library** that both `execute_plan.py` and any
   curl-based caller import. Hooks stay as defense-in-depth on direct curl.
2. **Make `execute_plan.py` shell out via curl** so hooks fire. Simpler but
   uglier — gives up `requests` features like connection pooling.

Recommend (1). Tracks alongside Phase 0.5 in priority but is its own item.
